# 明烛体系诊断报告 v2.1

> 基于 AgentBench、SWE-bench、WebArena、ToolBench、GAIA 等业内 agent 评测标准，
> 以及 Confident-AI 总结的 agent 失败模式，对明烛体系进行系统诊断。
> 诊断日期：2026-06-26

---

## 一、诊断方法

### 业内评测维度（来自搜索）

| 评测项目 | 核心评测维度 | 明烛是否覆盖 |
|----------|-------------|-------------|
| **AgentBench** (THUDM) | 8环境：OS/DB/KG/卡牌/横向思维/网购/家务/Web；指标：success rate、overall reward | ❌ 明烛无任何环境交互能力 |
| **SWE-bench** | 真实GitHub issue解决、代码生成+测试通过率 | ❌ 明烛无代码执行验证 |
| **WebArena** | 真实网页交互（点击/输入/导航）、任务完成率 | ❌ 明烛无浏览器工具 |
| **ToolBench/API-Bank** | 工具调用正确性、API参数填充、调用链规划 | ❌ 明烛无工具调用框架 |
| **GAIA** | 通用助手能力、多步推理、工具使用、文件处理 | ❌ 明烛无文件/工具处理 |
| **MT-Bench** | 多轮对话连贯性、上下文保持 | ⚠️ 明烛无多轮状态管理 |

### Agent 失败模式（来自 Confident-AI）

| 失败模式 | 说明 | 明烛是否存在 |
|----------|------|-------------|
| 工具调用错误 | 参数错误、调用顺序错误、调用不存在的工具 | N/A（无工具） |
| 幻觉 | 编造事实、伪造引用 | ⚠️ 红线检测有，但仅正则 |
| 目标漂移 | 执行过程中偏离原始目标 | ❌ 无检测机制 |
| 记忆失效 | 多轮后遗忘上下文 | ❌ 无记忆系统 |
| 过度自信 | 对错误结果高置信度 | ⚠️ 有谦逊评估，但仅关键词 |
| 循环调用 | 重复调用同一工具/人格 | ❌ 无检测 |
| 上下文超长 | 输入过长导致性能下降 | ❌ 无管理 |

## 二、明烛整体问题（7个）

### P0 严重（影响可用性）

#### 1. 无工具调用能力——不是真正的 agent
**现状**：明烛的8个人格全是"语言模型角色扮演"，没有任何工具调用（搜索、代码执行、文件读写、API调用）。在 AgentBench 的8个环境中，明烛得分为0。
**影响**：明烛只能"说"不能"做"，无法完成任何需要环境交互的任务。
**优化**：增加 `ToolRegistry` + `tool_call` 接口，让人格能调用工具。

#### 2. 无多轮记忆——每次对话从零开始
**现状**：`MingZhu.run()` 是无状态的，每次调用独立。`context` 参数需手动传入，无自动累积。
**影响**：无法维持连贯多轮对话，AgentBench 多轮任务全部失败。
**优化**：增加 `Memory` 类，管理短期（当前会话）+ 长期（跨会话）记忆。

#### 3. 无目标漂移检测
**现状**：`CognitiveCycle` 的执行阶段不检测目标是否偏移。执行过程中可能悄悄跑题。
**影响**：复杂任务执行到一半偏离目标，反思阶段才发现（甚至不发现）。
**优化**：执行阶段增加 `_check_goal_drift()`，对比当前输出与预见阶段的目标。

### P1 重要（影响质量）

#### 4. 评估器全是关键词正则——LLM-as-a-Judge 形同虚设
**现状**：`Evaluator` 的诚实性/谦逊/有用性/清晰度评估全是关键词计数（如"100%"扣15分）。`_llm_judge` 虽有接口但默认不接入真实LLM。
**影响**：质量评分可被轻易绕过（换个说法就检测不到），无法真实反映输出质量。
**优化**：接入真实 LLM-as-a-Judge，关键词规则降级为快速预筛。

#### 5. 路由是关键词匹配——无法处理语义级路由
**现状**：`route()` 用 `trigger in user_input` 做关键词匹配。"帮我分析下这个商业决策"能命中"商业"，但"这个买卖值不值得做"就命中不了乾断。
**影响**：路由准确率低，该调用的人格没调用，不该调用的调用了。
**优化**：增加语义路由（LLM判断或embedding相似度），关键词作为快速通道。

#### 6. 无失败恢复机制
**现状**：`_execute_persona` 失败后返回 `error`，但 `run()` 不重试、不降级、不换人格。
**影响**：单个人格LLM调用超时，整个任务失败。
**优化**：增加 `retry` + `fallback_persona` + `graceful_degradation`。

### P2 改进（影响成熟度）

#### 7. 无执行循环检测
**现状**：如果两个人格互相触发（A的输出触发B，B的输出又触发A），无机制检测和打断。
**影响**：理论上可能死循环（虽然当前架构是单轮并行，但协作模式collaborate可能触发）。
**优化**：增加调用深度限制 + 循环检测。

## 三、各子人格问题

### 乾断（qian_duan·逻辑决策）
- **问题**：无量化决策能力。声称"商业/财务/概率"，但无计算工具，所有"决策"都是语言推理。
- **业内对标**：AgentBench 的 DB 环境要求 SQL 查询计算，乾断做不到。
- **优化**：增加 `calculator` / `sql_query` 工具接口。

### 离明（li_ming·表达封装）
- **问题**：汇总逻辑是简单字符串拼接（`other_outputs`），无冲突检测。如果乾断说"做"艮守说"不做"，离明只是把两段话拼一起，不解决冲突。
- **优化**：增加冲突检测 + 仲裁逻辑。

### 巽风（xun_feng·搜索调研）
- **问题**：**最严重**——名为"搜索调研"人格，但无任何搜索工具。完全是"假装在搜索"。
- **业内对标**：WebArena/ToolBench 要求真实工具调用，巽风得分为0。
- **优化**：接入 `web_search` / `web_reader` 工具，让巽风真的能搜。

### 震造（zhen_zao·代码工程）
- **问题**：无代码执行验证。生成的代码不跑、不测，无法保证可运行。
- **业内对标**：SWE-bench 要求代码通过测试用例，震造做不到。
- **优化**：接入 `code_execute` / `test_run` 工具。

### 艮守（gen_shou·安全审查）
- **问题**：红线检测是正则，可被绕过。"我百分百确定"能检测，"我敢打包票"就检测不到。
- **优化**：接入 LLM 安全审查 + 对抗性测试集持续验证。

### 坤载（kun_zai·管理协调）
- **问题**：`collaborate` 方法存在但与 `run` 割裂，两套调度逻辑。且无任务拆分后的子任务追踪。
- **优化**：统一调度入口，增加子任务状态机。

### 坎观（kan_guan·观察者）
- **问题**：观察报告依赖LLM生成，无结构化指标。无法量化"观察质量"。
- **优化**：增加结构化观察指标（覆盖率、盲点数、置信度校准）。

### 兑泽（dui_ze·创造灵感）
- **问题**：无发散思维工具（如随机词、SCAMPER等技法），纯靠LLM自由发挥。
- **优化**：增加创意技法模板，结构化引导发散。

## 四、优化优先级

| 优先级 | 问题 | 优化项 | 工作量 |
|--------|------|--------|--------|
| P0-1 | 无工具调用 | ToolRegistry + 3个基础工具（search/read/calc） | 大 |
| P0-2 | 无多轮记忆 | Memory 类（短期+长期） | 中 |
| P0-3 | 无目标漂移检测 | CognitiveCycle 增加 drift 检测 | 小 |
| P1-4 | 评估器太弱 | 接入真实 LLM-as-a-Judge | 中 |
| P1-5 | 路由太弱 | 语义路由（LLM判断） | 中 |
| P1-6 | 无失败恢复 | retry + fallback | 小 |

本次优化聚焦 P0-3（目标漂移检测）和 P1-6（失败恢复），因为这两个工作量小、收益高、且不依赖外部LLM接入。P0-1/P0-2/P1-4/P1-5 需要接入真实LLM和工具系统，列为下一步。
